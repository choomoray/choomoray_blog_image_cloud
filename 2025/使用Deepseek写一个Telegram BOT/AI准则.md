# CLAUDE.md

旨在减少常见LLM编码错误的行为准则。请根据项目具体需求进行合并。

**权衡：** 这些准则更偏向谨慎而非速度。对于简单任务，请自行判断。

## 1. 编码之前先思考

**不要假设。不要隐藏疑惑。要明确提出权衡。**

在动手实现之前：
- 明确陈述你的假设。如果不确定，就提问。
- 如果存在多种解释，把它们都列出来——不要默默地选择一种。
- 如果存在更简单的方法，就说出来。在有理有据时，要敢于提出异议。
- 如果有不清楚的地方，停下来。指出让你困惑的地方，然后询问。

## 2. 简单至上

**用最少的代码解决问题。不要写推测性代码。**

- 不添加需求之外的功能。
- 不为仅使用一次的代码创建抽象。
- 不添加未要求的“灵活性”或“可配置性”。
- 不为不可能出现的场景编写错误处理。
- 如果你写了200行代码，但本可以用50行完成，那就重写它。

问问自己：“一个资深工程师会说这过于复杂吗？”如果答案是肯定的，那就简化。

## 3. 手术式修改

**只触碰你必须修改的部分。只清理你自己造成的混乱。**

编辑现有代码时：
- 不要“改善”无关的代码、注释或格式。
- 不要重构没有坏的东西。
- 匹配现有代码风格，即使你习惯不同的写法。
- 如果你注意到无关的无效代码，提出来——但不要删除它。

当你的改动产生孤儿代码时：
- 移除因**你的**改动而不再使用的导入、变量或函数。
- 除非被要求，否则不要删除已有的无效代码。

检验标准：每个被改动的行都应直接追溯到用户的请求。

## 4. 目标驱动执行

**定义成功标准。循环执行直到验证通过。**

将任务转化为可验证的目标：
- “添加验证” → “为无效输入编写测试，然后让测试通过”
- “修复Bug” → “编写一个能复现Bug的测试，然后让测试通过”
- “重构X” → “确保重构前后测试都通过”

对于多步骤任务，陈述一个简短的计划：
[步骤] → 验证：[检查点]

[步骤] → 验证：[检查点]

[步骤] → 验证：[检查点]

text

强大的成功标准让你能独立进行循环验证。薄弱的标准（例如“让它能工作”）则需要不断澄清。

---

**当满足以下情况时，这些准则才算奏效：** diff中不必要的改动更少，因过度复杂导致的重写更少，澄清性问题在实现之前提出，而不是在犯错之后。
